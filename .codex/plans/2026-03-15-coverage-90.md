# Piano Copertura 90% Totale `app/**`

## Summary
- Obiettivo: portare la coverage Vitest dal `7,44%` attuale ad almeno `90%` su `statements` e `lines` calcolati sull'intero `app/**`.
- Ambito deciso: tutto `app/**`, con refactor mirato consentito, test DOM con `Vitest + jsdom + React Testing Library`, nessuna esclusione pragmatica di route/UI.
- Le uniche esclusioni ammesse sono generated, type-only, barrel e wrapper statici realmente non significativi.

## Success Criteria
- `npm run test:coverage` termina con:
  - `statements >= 90`
  - `lines >= 90`
- `functions` e `branches` vengono alzate il più possibile, ma non sono gate a `90`.
- `npm run test`, `npm run lint`, `npm run typecheck` devono passare.
- Nessun cambio ai contratti runtime pubblici di route o component props, salvo refactor interni per testabilità.

## Test Cases and Scenarios
- Route/API: body valido, body invalido, query invalida, request senza body, handler che lancia, response status/code/body coerenti.
- Services/provider: configurato/non configurato, success path, provider error, `401` con retry, refresh token scaduto, partial failures.
- Chat/memory persistence: create, get, append, delete, search, fallback quando summary/embed falliscono, auth assente.
- Assistant tools: tutti i tool `execute`, validazione schema, mapping success/error, side effects asincroni, tool sconosciuto.
- Transport/provider: connessione, eventi in ordine inatteso, cleanup, listener rimossi, payload malformati.
- `useVoiceChat`: start/stop listening, wake word -> connected, disconnect -> wake word, mute, delete chat, inactivity timeout, restore/fetch chat iniziale, error surface.
- Timer: start, pause, resume, stop, finish automatico, formatting.
- UI/design: render base, callback, stato vuoto, stato loading/error, rendering condizionale, componenti assistant/calendar/tasks/timer.

## Assunzioni e Default
- Il target vincolante finale e `90%` su `statements` e `lines`.
- `branches` e `functions` non sono gate a `90`.
- Il perimetro resta l'intero `app/**`.
- Sono ammessi refactor interni mirati, non redesign estesi.
- Le esclusioni restano minime; se un file sembra non testabile, si preferisce estrarre logica pura prima di valutare una nuova esclusione.
